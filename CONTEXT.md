# Archie

A static-publishable, multi-media exhibit annotation platform: OSD + Annotorious deep-zoom image annotation, audio/video annotation, W3C WADM-compliant, IIIF Presentation 3 on disk, authored in a browser Studio and published as a static site to GitHub Pages. v4 design doc is the trunk; prior-art survey is cited as evidence.

## Locked frames (user, non-negotiable)

- OSD + Annotorious
- Two surfaces: (CMS Gallery + Exhibit-making UI) = **Studio**; (Exhibit Gallery + published Exhibit UI) = **Viewer**
- Multi-media: audio, video, image
- Multi-object exhibits
- Annotations at **both** project level and image level
- Linkable + navigable annotations
- WADM (W3C Web Annotation Data Model)
- Web-publishable: **GitHub Pages**, standalone, no server

## Language

**Library**:
Top-level container for one project; on disk a directory/zip; IIIF `Collection`. Holds many Exhibits plus project-level metadata and notes.
_Avoid_: Project (v4 uses Library; "project" is overloaded — anvil meant the working dir, the brief means the Library).

**Exhibit**:
One published narrative artifact; IIIF `Manifest`. Self-contained: owns its own Objects, media, annotations, and narrative pages.
_Avoid_: Story, collection.

**Object**:
One media item inside an Exhibit (image / audio / video / **map** / external embed); IIIF `Canvas`.
_Avoid_: Item, asset, hotspot.

**Map** (medium — 2026-06-18 grilling; geo-annotation extension, DESIGN.md):
A fourth Object **medium** beside image / sound / video — an Object whose surface is a slippy-map **basemap** (XYZ/raster tiles), mounted on the SAME OSD deep-zoom Canvas as an image. **Medium ≠ editor:** a Map reuses the image Canvas (unlike sound/video, which route to the AvEditor) — the medium distinction is about *meaning and affordances* (geographic CRS, pins-not-regions, required attribution, framed region), not which renderer it reuses. Carries a `tileSource` descriptor (tile template + zoom range + the **framed region**) the way sound/video carry `duration`. Geo-Notes (**Box/Outline regions only — NO pins**, 2026-06-18 user decision) target its Canvas, drawn with the existing Box/Outline tools (ADR-0011), shape vocab unchanged (Q-1 rect + polygon); the **geographic anchor (lng/lat) is the source of truth, the image-pixel selector its derived projection** (geo-truth — re-framing the extent keeps regions nailed to the Earth; the pixel selector recomputes via `lngLatToPixel`). The author **frames the region of the map shown** — the *Map extent* (below) — and may further frame sub-regions via the ADR-0005 camera (narrative).
_Avoid_: basemap-as-its-own-top-level-noun (a Map is an Object — ADR-0001); "map layout" (arrangement-of-objects is the orthogonal *spatial arrangement* axis); calling a map an "image" (medium is the load-bearing distinction).

**Map extent** (ADR-0015, 2026-06-18):
A Map's **bounded geographic region** — `tileSource.bounds = [west, south, east, north]` — that *is* the map: the OSD pixel raster is the world sub-rectangle covering those bounds, and the frame is **absolute** (the reader cannot pan past it; the author sizes the frame for the context they want). The extent is the Map's finite pixel dimensions (the geographic analogue of an image Object's `width`/`height`) and the unit that makes a basemap **bakeable for offline** (ADR-0010/ADR-0004 — a whole-world basemap is unbakeable; a bounded region is). The ADR-0005 **camera** (`Section.start`, extended to a geo-fragment) frames sub-regions *within* the extent for narrative; it does not replace the extent.
_Avoid_: confusing the *extent* (the map's coordinate space, absolute) with the *camera* (a view within it, ADR-0005); a whole-world Map (rejected — ADR-0015).

**Note**:
A single W3C WADM `Annotation`. Targets a Library, Exhibit, Object, image region, time range, or a **geographic region on a Map** (a Box/Outline anchored by geo-truth — no point/pin form).
_Avoid_: Annotation (in UI), comment.

**Reading** (renamed from "Layer"/"Lens" 2026-05-26 — Frame C; the old term was one word doing two jobs, a *stroad*):
A curated, named, **mutually-exclusive interpretive pass over a source** — the author's declaration of *one way to read it* (e.g. a *cipher* reading vs a *hoax* reading of the same Voynich folio). The reader **switches** between Readings; toggling re-frames the canvas. **Enables competing, collaborative readings of the same Object** — different scholars' interpretations *coexist* rather than being merged into one (the structural home for legitimate scholarly disagreement; see the collaboration connection below). A Reading **IS an IIIF `AnnotationPage`** (per Object) grouped by an **`AnnotationCollection`** (per Exhibit — the Reading's identity `{name, description, colour}`); a Note carries **one optional `reading` id** (mutual exclusivity, Q5) → pure IIIF viewers (Mirador) get real toggleable readings free. **Apparatus / reference notes (paleography, codicology, material) are NOT Readings — they are Tags** (Frame C: the additive, note-pane job split off the overloaded old "Layer").
_Avoid_: "Layer"/"Lens" (superseded, overloaded); group; conflating with **Tag** (Tag = additive, per-note, discovery, note-pane) or **Section** (Section = sequence / IIIF `Range`; Reading = interpretation / IIIF `AnnotationPage`).

**Tag**:
A lightweight ad-hoc label on a single Note; a WADM body with `purpose: tagging`. Visitor payoff = **cross-cutting discovery/retrieval**, NOT framing: a flat **filter chip** ("show only notes tagged `signature`") with no curation, no editorial weight, no legend, no visual identity, **additive** (stack freely). The deliberate inverse of a **Reading** (Q1, 2026-05-26). **Now also the home for apparatus / reference strata** (paleography, codicology, material) under Frame C — these are *additive note-pane filters*, not exclusive canvas Readings. (This finally gives Tags a clear purpose, fixing their own underuse.)
_Avoid_: Keyword; calling it a synonym for Reading.

**Spatial arrangement** vs **Reading mode**:
The two orthogonal axes a "layout" conflates. *Spatial arrangement* = how objects sit (one / many / side-by-side). *Reading mode* = how the visitor moves (click-driven vs scroll-driven, prose-led vs object-led). Slideshow is a reading-MODE of the Gallery arrangement, not its own template.
_Avoid_: "layout template" as an undifferentiated bucket.

**Section**:
One ordered unit of an Exhibit's narrative; an IIIF `Range`. A **Single**-template Exhibit = one Section; a **Narrative**-template Exhibit = N ordered Sections (prose is the spine). Keeps Single ≠ "narrative with one section."
A Section is a **self-contained reading beat** (resolved 2026-05-25, the "third layer" model — Q-N below): `{ objectId, start, prose }`. `start` = the camera target on that object, a **media fragment** — `xywh=…` for an image object, `t=start,end` for an AV object (same grammar as a Note's selector). The spine may **switch objects** across sections (map → photo → recording); single-object is the degenerate case where every Section shares one `objectId`. A Section is **independent of Notes** — Notes are a separate annotation layer; a Section that wants to point at a Note does so via a ⌘K **link in its prose** (the linkability mechanism), NOT a structural reference. While a Section is active the Viewer shows that object's markers (progressive-reveal §122 deferred to v1.1, redefined as "markers whose region the narrative has passed"). Authoring convenience: "**add section from a Note**" seeds a new Section's `start` + prose from an existing Note (gives a note-walk's ease inside the third-layer model). **Authored in the EDITOR, not a separate screen** (2026-05-25): a Section's camera is FRAMED on the object's canvas (the same gesture as a Note's geometry — draw a box / set in-out), the object chosen via the rail, with prose + spine-order in a sidebar Narrative panel beside the notes list. The overview-as-canvas is for VIEWING/arranging the exhibit, NOT for authoring sections — splitting them divorces the spine from the canvas the camera is framed on.
_Avoid_: chapter, stop, scene. Do NOT model a Section as a reference to a Note (the rejected "tour-of-notes" model — it couples the narrative layer to the annotation layer and can't focus a non-Note region).

**Studio**:
The authoring app. Browser SPA. Writes working state, exports zips, runs the publish build.

**Viewer**:
The read-only "one smart hall" shell (§216). ONE build; its data source is the variable, auto-detected at runtime (2026-05-27): a baked published tree present → **hosted Viewer** (the deployed published site — the original meaning); absent → **portable Viewer** (an empty hall that opens a `.archie.zip` a recipient was handed, no Studio/server — the recipient-reads gap). The seam is one file (`apps/viewer/src/published.ts`): hosted fetches the tree, portable reads it from `ZipFilesystem.fromZip`.
_Avoid_: treating the portable Viewer as a separate app/build (it is a mode of the one shell); calling a `.archie.zip` reader an "import" in UI copy (it OPENS a Library — see **Library**).

**hosted Viewer** / **portable Viewer**:
The two data-source modes of the one **Viewer** (above), not two apps. **Hosted** = baked tree at a URL (current behaviour; never shows file-opener chrome). **Portable** = opens a `.archie.zip` in-browser. **"Open another library"** (escape affordance, anchored in persistent chrome so the single-exhibit collapse can't trap the reader) shows whenever a library is loaded — hosted OR portable (revised 2026-05-27, user override of the original portable-only rule; see ADR-0008).

**Gallery**:
The **Library-level** published index of Exhibits — the published site's landing ("exhibit gallery") and the Studio's library-browse ("cms gallery"), one data source / two renders. Source = a generated `exhibits.json` (a build-time projection of the Library; the survey's MAJOR greenfield gap — nobody emits a multi-exhibit index). Distinct from an IIIF Collection (rendered browse surface, multi-exhibit).
_Avoid_: using "Gallery" for the within-exhibit object grid (that is **Grid**).

**Grid**:
A within-Exhibit **layout** showing an Exhibit's Objects as a browsable thumbnail grid. (Was misnamed "Gallery" in v4's layout set.) Slideshow is a *mode* of Grid.
_Avoid_: Gallery (reserved for the Library-level index).

## Relationships

- A **Library** contains one or more **Exhibits** (1→N)
- An **Exhibit** owns one or more **Objects** (1→N) — objects are NOT shared across Exhibits (see Flagged ambiguities)
- A **Note** targets exactly one of: Library, Exhibit, Object, region, or time range
- A **Layer** groups many **Notes** across Objects within an Exhibit (cross-cutting)
- A **Tag** belongs to one **Note**
- The **Studio** writes what the **Viewer** reads; both consume the same on-disk WADM/IIIF files

## Decisions (resolved in grilling; ADRs pending — user deferred writing them to end)

- **Object ownership = exhibit-nested** (2026-05-24). Objects + their Notes live inside each Exhibit; no cross-exhibit reuse. See Flagged ambiguities.
- **Shape vocabulary = rect + polygon only in v1** (2026-05-24). The two that round-trip losslessly through stock `W3CImageFormat`. Ellipse/freehand deferred to v1.1 behind a custom svgpath parse/sanitize/serialize module that intercepts non-rect `SvgSelector` before the broken adapter branch. Evidence: `wadm-roundtrip` verdict — Ellipse→NaN, Path→curves-stripped, silent, in real Chrome. Donor: svgpath (runtime-verified), points-on-path.
- **`@render` = 3 layers** (2026-05-24). `@render/core` (pure TS: selector types, URL serialization, IIIF resolution, geometry/hit-test, OPFS adapter, scroll-sync controller — donor logic adopted/laundered) → `@render/mount` (vanilla mount fns wiring OSD + Annotorious-core + Wavesurfer) → thin per-framework adapters (<500 LOC each; budget doubles as logic-leak detector). Premise rejected: the valuable shared things are logic, not components.
- **Framework = Svelte everywhere** (2026-05-24). Studio = Svelte SPA (adopts anvil's editor shell/AnnotationForm/List/NarrativeEditor/PublishDialog/keyboard-registry/lifecycle as RUNNING code). Viewer = Astro + Svelte islands (adopts annomea's 3-state pane + popup+drawer). One adapter `@render/svelte`, zero `@render/react`. Read-UI donors are all Svelte; Juncture is Vue → its value is PURE (→ core), neutral on the choice. Rejected React-islands split: would re-derive every Svelte read-side donor + pay an unpaid-for two-adapter + hiring/third-surface tax.
- **Storage = OPFS default + FSA folder opt-in** (2026-05-24). Zero-friction OPFS working store by default; power users "link to a folder" (FSA, Chromium) for a git-native, visible copy. zip is the universal bridge. Two backends behind the Filesystem seam.
- **Tiling = single-JPEG + external IIIF (v1); OffscreenCanvas DZI slicer (v1.1)** (2026-05-24). v1: OSD `type:'image'` single responsive JPEG (~6000–8000px); giant images via pasted external `info.json`. NO wasm-vips (empirically has no `dzsave`; ~13–20MB binary blows the 240KB budget). v1.1 in-browser tiling = OffscreenCanvas worker pyramid, NOT libvips-WASM.
- **AV transcripts = WADM `supplementing` body, import-only v1** (2026-05-24). Author supplies WebVTT/SRT; each cue → Note with `motivation:supplementing` at `t=start,end`. No client-side ASR (Whisper is server-side). Build the VTT→`TextualBody` adapter. Time Notes = `FragmentSelector t=start,end` / `PointSelector t=`; AV source typed `Sound`/`Video`.
- **Navigation contract = full in v1** (2026-05-24). `fitBounds` on EVERY select (sidebar, narrative-prose link, `#/a/<id>` deep-link); polygon→bbox to feed OSD `fitBounds` (`goToTarget` is rect-only); popup/drawer re-anchors live on pan/zoom. Closes annomea's documented root-cause (fitBounds called zero times). Adopts anvil's fitBounds wiring into `@render/core`.
- **Annotation mutability = append-only + version DAG, BOTH provenance (A) and async-zip collab merge UI (B) in v1** (2026-05-24, user override of strip-it recommendation, confirmed intentional). Driver: scholarly citation integrity + genuine async multi-author workflow.
- **Merge = version-DAG causality, three-way-without-git** (2026-05-24). Each Note carries `{logicalId, version:int, parent, modifiedAt, lastEditor:clientId}`; edit bumps version, append-only keeps the parent chain. Merge per logicalId: same version → no card; one side fast-forwarded (parent matches) → no card; both advanced from common ancestor → manual conflict card. Wall-clock `modifiedAt` is ONLY an in-card tiebreaker (clock skew makes timestamp-LWW a data-loss bug). Merge-base computed by walking the parent DAG → three-way merge UX (ancestor + per-side diff) without git.
- **WADM serialization = heads-projection + history sidecar (log + projection)** (2026-05-24). Canvas AnnotationPage that viewers load holds ONLY head version(s) — plural after unresolved concurrent merge (both overlays = honest degradation). Full version chain persists per-logicalId as WADM AnnotationPages at `annotations/history/{logicalId}.json`, indexed by `annotations/history/index.json` (logicalId→URL; merge's load target; per-note pages are citation-dereference targets). Head links out via `archie:hasHistory` (→ history page) + `prov:wasRevisionOf` (→ prior version id); pure consumers ignore both. Versioned id = `{logicalId}/v{n}` (resolvable path, static-host-friendly). **Three-tier interop contract:** pure WADM consumer → current state; PROV-aware consumer → history; Archie viewer → full DAG; none broken. A "compile heads page" step in the Studio publish pipeline (alongside tiling) is a pure idempotent function of the append-only log. This is the unifying SPINE: append-only log → version-DAG merge → heads projection, one pattern at three boundaries.

- **Publish = zip-primitive + per-host adapters** (2026-05-24). Build produces a zip (the architectural primitive); each host is a thin adapter over the zip contents. GH Pages adapter (~200 LOC) uses the **GitHub Contents API** (not isomorphic-git — "replace this tree" matches), fine-grained PAT (contents:write + pages:write, 90-day). Token NOT persisted initially (paste-each-publish; WebCrypto PBKDF2→AES-GCM only if requested). First-class "Connect to GitHub" walkthrough opens the token page pre-filled with scopes. zip is the universal zero-auth default. "No server" lock = published artifact has no runtime; pushing to api.github.com is talking to a server, not running one (same category as fetching museum IIIF).

- **Orphan gaps (2026-05-24).** Scheduled by GATE not dependency. (1) **EXIF orientation = original-source + display-master-projection, v1.** Keep original untouched (provenance); ingest generates a normalized display-master derivative (derived bucket alongside tiles, regeneratable, lives under `.archie/cache/`-style path, doesn't ship to GH Pages unless "include source for citation" opted in). Annotations target the display master — ZERO orientation-awareness in the coord layer (this is why it beats non-destructive forever). Lossless rotation for pure rotation/flip (all 8 orientations qualify), re-encode PNG/HEIC. Provenance metadata per object: original path+SHA+EXIF, master path+SHA, transform applied. Gate: **8-orientation test-fixture set on every build before first public exhibit.** Fits the spine (source/projection/build/consumer). (2) **Empty/error/loading states = v1** (public exhibit can't ship blank; genuinely resists the spine — UI judgment). Gate: audited before public Viewer ships. (3) **Schema migration = v1.1 runner, but stamp a version field in v1** so v1 files are migratable (tldraw named-migration pattern; fits spine as versioned-files→migrated projection). Gate: discipline established before first schema change ships. (4) **Overlay contrast = v1.1** (perceptual, resists the spine). Gate: before first institutional pilot.

- **Layers = IIIF AnnotationPages, v1** (SUPERSEDES "per-note membership" 2026-05-24, reversed 2026-05-26 after the "layers should work the way annotation pages do for IIIF" grill). **Reason for reversal:** the per-note `archie:layers: string[]` string model required Archie-specific filtering, so pure IIIF viewers could only "show all." Modelling a Layer as an **`AnnotationPage`** (per-Canvas) grouped by an **`AnnotationCollection`** (per-Exhibit, the lens identity) is IIIF-native: pure viewers get **real toggleable layers for free**. The codebase already reserved the hook — `W3CAnnotationPage.partOf` is commented "Layer membership (IIIF AnnotationCollection reference)" (`render-core/src/wadm/types.ts:122–130`), so §92's string model was a v1 shortcut, not a deep commitment. **Original §92 objection still honoured:** v4 grouped *canvases*; this groups *notes* (each per-Canvas page holds one lens's notes), just via the IIIF-native serialization. **Membership = mutually exclusive, one Note → one lens or none** (Q5) — required because IIIF puts an annotation in exactly one page's `items`; multi-membership (duplication/hybrid tax) deferred to v1.1 as a deliberate, knowing step. **Heads projection changes** from "one page per Canvas" to "N pages per Canvas, partitioned by lens + a default page for unlayered notes" — still a pure idempotent function of the append-only log (the spine pattern §86 holds; the projection just partitions). Tags (`purpose:tagging` → per-note chip filter) unchanged. **ADR-0007** (written 2026-05-27; amends ADR-0003; reverses this lock — user-directed).
- **Layout v1 set = Single + Grid + Narrative** (2026-05-24). Slideshow = a **Grid** *mode* (not a template). Narrative = prose-spine + N ordered Sections (IIIF Ranges), click-activated via the Q8 nav contract (the *navigable* narrative; nav engine already v1). Cheap = prose pane (Svelte island + Q8 link contract) + canvas pane (same OSD mount) + Range→section binding + click-to-activate. v1.1 = Scrollytelling (passive scroll-spy/pinning half, inherits Narrative) + Compare (synced dual-canvas, different spatial arrangement). Honest: narrative-in-v1 ADDS UI scope (not a free swap), but matches the locked "exhibit = narrative." a11y disposition accepted as-is: keyboard nav + alt-text-as-WADM-body + focus mgmt v1; overlay-contrast v1.1.
- **Library-level Gallery + `exhibits.json` = v1** (2026-05-24). The Library is the unit of publication: one Library → one published site = a Gallery landing + N Exhibits. `exhibits.json` is a generated build artifact (projection of the Library) — the survey's MAJOR greenfield gap, ours to build. Studio library-browse and the published Gallery share this source. Required by the locked "exhibit gallery + published exhibit ui" and assumed by intra-Library linking (needs the Library-wide index).

- **Linkability = intra-Library structured + cross-Library deep-link** (2026-05-24; closes the under-grilled half of locked frame "linkable"). "Linkable" has two scopes v4 conflated. **Intra-Library:** Note links target any Note/region/Exhibit in the same Library; ⌘K stores a structured reference `{libraryId, exhibitLogicalId, noteLogicalId|rangeId|xywh}` + a display-URL projected at publish (author-time resolvable, publish-time validatable, auto-updating if target path changes). **Cross-Library:** a link IS a Q8 deep-link URL someone else's Viewer serves — reuse the deep-link contract, do NOT build a separate system; ⌘K offers "paste URL" + recent-targets (no search); durability = open web's (repairable via stored logicalId, never self-healing); NO link-checking/federation/registry (server-shaped, ruled out). **Anno-index = BOTH per-Exhibit (Viewer read-time) AND Library-wide (Studio author-time)** — both projections of the log. Pin a stable published URL grammar (`/{exhibitSlug}/` + deep-link fragments) so cross-Library links are repairable. New Viewer UI: deep-link for a non-loaded Exhibit → "open it?" affordance, not silent same-exhibit treatment.

## Architectural through-line (page one of the arch doc)

**Rule: define the authoritative source, project thin.** Every major decision resolved to source-of-truth + thin derived projection/adapter:

| Boundary | Source of truth | Thin projection / adapter |
|---|---|---|
| Rendering | `@render/core` (pure TS) | per-framework adapters (<500 LOC) |
| Storage | Filesystem seam | OPFS / FSA backends |
| Annotations | append-only log | projected head version(s) |
| Merge | version-parent DAG | conflict-card view (computed merge-base) — **interactive/stateful: the one row where "thin" strains hardest, highest impl-risk-per-line** |
| WADM export | the log | heads page + history sidecar |
| Publish | the zip | per-host adapters (~200 LOC) |
| Deep-zoom | source image | tile pyramid (build step) |

**Orphan gaps resist this pattern** (correctness/polish/durability/perception, not data-flow) → schedule by GATE not dependency (see Decisions). **Collaboration's non-spine UI/workflow cost** (~4–6 weeks) is the dominant remaining cost and is not derivable from locked frames.

## UX decisions (end-to-end journey grill)

- **First-run = Demo/Real split by entry path, NOT a storage prompt** (2026-05-24). Landing has two doors. **"Try a template"** → explicit *Playground* mode, honest banner "Playground — nothing is saved. Start a project to keep your work." (ephemeral OPFS; preserves 60s-to-first-annotation). **"Start a project"** → a real, persisted Project. Conversion: one-click "Save this playground as a project" at first-value (first export/publish, ~5 min). Kills the nudge strategy (nudges = symptom of a default mismatching intent).
- **Persistence = three configs behind the Q4 Filesystem seam, selected by capability+intent; user-facing model is ONLY Playground vs Project** (2026-05-24). Never expose browser capability to the user. **Chromium Project** → `FsaFilesystem` (pick a folder once, autosave in place — the git/GH-Pages on-ramp). **Non-Chromium Project** (Firefox/Safari lack writable FSA) → `DownloadFilesystem` (OPFS working copy + the **zip IS the canonical file**: explicit Save = download `.archie.zip`, Open = pick it — the "Word-doc 2003" model). **Both browsers' Try** → ephemeral OPFS. This REFINES Q4: real-project default is the folder (Chromium) / zip-as-file (else); OPFS demoted to playground + non-Chromium working store. Capability ceiling as high as the browser allows, graceful floor everywhere. **Gate/validate:** non-Chromium users must grok zip-as-canonical-file — the failure mode is "Start a project again" instead of "Open"; mitigations = Open as prominent as Start on landing + a **recent-projects list surviving sessions** (metadata in OPFS/localStorage — a fine use of invisible storage; it's metadata, not content) + prominent habit-forming Save (keyboard shortcut + "unsaved changes" indicator).

- **Authoring workspace = Library screen + Exhibit canvas-workspace (two scales)** (2026-05-24). **Library** = the Project's home, a list/grid screen for project-level work (browse/organize/pick Exhibits, metadata, publish, export) — deliberately list-UI, visually distinct from canvas. **Exhibit** = anvil's canvas-centric workspace (ADR-0008) at TWO scales sharing the canvas+sidebar pattern: **overview** (object thumbnails in reading order + section dividers; sidebar = exhibit-level props/sections/narrative) ↔ **object** (anvil deep-zoom annotation surface; sidebar = WADM form). Same zoom metaphor both scales (1a). **Media = a cross-cutting panel** accessible from both scales + inline attachment in the WADM form — NOT a separate screen (media is always in service of something else). **Breadcrumb** `Project › Exhibit › [Object N]`, always shown + clickable — the most-traveled nav. **Validate:** does overview-as-canvas feel like a canvas vs a list-pretending-to-be-one? Fallback = 1b (explicit overview/object mode toggle). Honest retreat path, not v1 commitment to the clever version sight-unseen.

- **Collaboration UX = silent DAG-merge + summary panel + inline resolution** (2026-05-24). Author flow: "Share a copy for review" (export zip + message) / "Import changes" (pick returned zip). On Import, the Q10 version-DAG silently fast-forwards everything mergeable; a **summary panel** ("Synced 42 notes from Alice · 3 need your decision · [Review][Later]") answers "am I done?" + "what happened?" in one line. **Review** drops into the first conflicted note *in the normal workspace* — a conflict-card variant of anvil's WADM form (ancestor + per-side, your Q10 three-way), "Next" advances. **Later** → conflict badges on notes + a persistent "N unresolved" chrome indicator. Zero new *editing* surfaces; novelty = just the summary panel + conflict-card form variant. **Identity** = local display name, prompted on first Import ("what should Alice see as your name?"), not at launch; skip → "Anonymous"/clientId, gentle re-prompt. Validation gate: does a non-technical author grok the SUMMARY PANEL unprompted (conflict cards are well-trodden — Git/Word/Figma)?

- **Editing happens AT the locus; selector dimensionality = the medium's** (2026-05-26, ADR-0006; grilled). REFINES §121's "zero new editing surfaces" → *the editing form has ONE definition; it ANCHORS to the marker (author = an editable canvas popover) or RENDERS inline (reader = annomea popup)*. The sidebar list stays as the scannable **nav index** (click → fitBounds §83 → popover; draw → popover opens at the new marker, so *add* = draw→type). **Note = point** (locus = marker → popover); **Section = reading beat** (locus = the reading column → prose edited in the sidebar narrative spine, mirroring the reader's prose column, + the same ⌘K; camera framed on canvas per ADR-0005). EXTENDS §82 (AV-temporal-only) → **selector dim matches medium dim**: image `xywh=` (Annotorious rect) · audio `t=` (**WaveSurfer** regions = the waveform locus) · video **`xywh=&t=`** spatiotemporal (frame-box + timeline). **Staged:** Wave 1 = note popover + ⌘K + section-spine ⌘K + audio-WaveSurfer; Wave 2 = video spatiotemporal (combined selector in core + frame-draw + timeline). **Popover mitigations** (or fall back to a pinned sidebar inspector, the 1a/1b retreat): offset off its marker · draggable · dismiss/detach on pan-zoom (not pixel-chase).

- **Reading default = layout-dependent** (2026-05-24). ### UX design philosophies (cross-cutting — the real synthesis; test future UX decisions against these)

Six principles emerged across UX-Q1–Q7 and now bind the decisions. Write them in the design doc *separate from per-feature specs* — they're what makes the decisions cohere rather than read as independent picks. A future decision that **violates** one is a signal to argue the violation explicitly or reconsider.
1. **Respect user intent declarations** (Q1 playground-vs-project · Q3 author's layout choice · Q5 reading-default-by-layout · Q7 Library framing). A published Exhibit looks like what the author *chose to make*; defaults honor declared intent, never override. → the layout-picker must make reading-intent legible (a sentence per layout).
2. **Surface decisions at the moment they acquire meaning** (Q1 storage-at-first-value · Q4 identity-at-first-send-back). Not as up-front setup tax.
3. **Precision in, escape out** (Q6 deep-link land-in-context · Q2 zip-as-file portability). Meet the user where they arrived; give ways *out* to context. Anti-pattern: orient-first / wrap-everything-in-onboarding.
4. **Inverse for top-level entry points: invite in, don't escape** (Q7 Gallery — the top context has no precision to respect; the move is to draw the visitor down into exhibits).
5. **One mental model exposed; configurations hidden** (Q2 "Project" over three persistence configs; never expose browser capability).
6. **Adopted patterns ship clean; inventions get prototype gates** (the Q4-onward discipline).

### Readings & Tags — purpose made legible (2026-05-26, grilling; closes the "underused, unclear purpose" symptom)

> **TERMINOLOGY (Frame C, 2026-05-26 — locked):** "Layer" is RETIRED. Diagnosis: today Layer and Tag both collapse to the same observable outcome — "filter notes" — so neither author nor visitor feels a reason to reach for one over the other, and both rot. §92 locks BOTH for v1 (not relitigated); the fix is to make each one's distinct purpose **legible in the UI**, author-side and visitor-side. - **Visitor payoff split (Q1).** **Layer = a named *lens* with visible identity** (framing — "a way to read this exhibit"); **Tag = a flat chip for cross-cutting discovery** (retrieval, no curation/identity). The load-bearing consequence: a Layer gets visual identity + a legend; a Tag never does.

- **Authoring scale split (Q3, decisive user rule 2026-05-26): "Layers are overarching UX; Tags are per-note UX."** A **Layer is created/named/described/coloured/ordered at the OVERARCHING (Exhibit) level** — a deliberate, top-down curatorial act in an Exhibit-level **Lenses panel** (overview sidebar, §119). The set is intentionally **limited** (curation = a small named set, not free growth) — so per-note assignment picks from the curated set; *adding to the set is an overarching act, never free-typed on a note*. A **Tag is authored entirely per-note** — inline create-on-type freetext on the note form (autocomplete from existing tags for convergence), no overarching surface, no identity. **The authoring scale mirrors the whole afterlife:** overarching/curated → framing/identity/canvas-legend; per-note/throwaway → discovery/flat/note-pane-chip. **Rejected (fork B):** "one label gesture + promote-to-lens" — makes Layer emerge bottom-up from per-note labels, which contradicts Layer = overarching UX. Rejected (original): per-note checkboxes as the layer-creation surface — wrong level (cramps an overarching act into a per-note control).
- **Data-model consequence (Q3+Q5 → keystone schema, ADR-worthy).** A Layer is an **Exhibit-owned `AnnotationCollection`** `{ id, name/label, description, colour, order }` (the lens registry); on each Canvas its notes form an **`AnnotationPage`** with `partOf` → that collection. A Note carries a **single optional `layer` id** (mutual exclusivity, Q5), which routes it into the matching per-Canvas page at projection time. `Canvas.annotations` becomes a **multi-element array** (one page per lens present + a default page); the mount layer loads pages and toggles per-page (today it flattens one page — `mount.ts:150`). A **Tag stays a bare per-note string** (`purpose:tagging` body, per-note chip). Hard to reverse (serialization + manifest + mount + migration), surprising (reverses §92), real trade-off (IIIF-native free pure-viewer layers vs a bigger build surface) → **ADR-0007** (written 2026-05-27, `docs/adr/0007-readings-as-annotationpages.md`).

### Exhibit / Library rights & metadata (2026-05-27, grilling — fields + Studio editing)

- **IIIF-native framing (Q1, locked).** Attribution / license / credits / provider are NOT Archie-invented fields — they **project to the IIIF-standard Manifest (Exhibit) / Collection (Library) vocabulary**: attribution → **`requiredStatement`** (`{label, value}` a conforming viewer MUST display), license → **`rights`** (a license URI — CC / RightsStatements.org), institution → **`provider`**, everything else → **`metadata`** (`{label, value}` pairs). Same move as Readings→AnnotationPage: author edits friendly inputs, the model projects thin so pure IIIF viewers (Mirador) show credit+license for free. **Undoes two hacks:** the Beinecke credit currently jammed into `library.summary` (`voynichCredits`) and the Bidar attribution jammed into `exhibit.summary` — both abuse `summary`; they move to `requiredStatement`/`provider`. **Studio must EXPOSE editing of all of these** (it currently exposes few/none) — see the Studio properties-editing branch below.
- **THREE levels, same fields, opt-in cascade (Q2 + extended 2026-05-27).** **Library (Collection) → Exhibit (Manifest) → Object (Canvas)** each carry the **same** rights vocabulary (IIIF puts `rights`/`requiredStatement`/`metadata` on a Canvas too). The Object level is often the *truest* provenance (each folio = Beinecke; a mixed exhibit = a Beinecke folio + a COMPOST photo). Inheritance is **OPT-IN, not automatic**, and **cascades Object → Exhibit → Library**: a level shows/publishes exactly its own values unless the author explicitly chooses "use the parent's value" for a field — no silent drift, DRY convenience on demand. **Placement (CONTEXT §119):** Library metadata → **LibraryHome** screen; Exhibit props → **Exhibit overview sidebar**; Object props → the **object editor sidebar** (beside the WADM form).
- **Field shapes (Q3, IIIF-simplified on the model → projected to full IIIF at publish):** `rights` = one license URI (Studio: a **license picker** of common CC / RightsStatements.org + "unspecified"; NOT free text — `rights` must be an approved URI) · `requiredStatement` = `{label, value}` (default label "Attribution"; the displayed credit) · `provider` = `{label, homepage?, logo?}` (→ IIIF Agent) · `metadata` = `{label, value}[]` · `inherit` = per-field opt-in flags (publish resolves to the parent's value). Replaces the `summary`-abuse hacks (`voynichCredits` on `library.summary`; the Bidar attribution on `exhibit.summary`).
- **Contributors (Q4, locked 2026-05-27) — a derived aggregate, NOT a cascaded field.** IIIF has no first-class contributor field, and contributors run *opposite* to the rights cascade: rights fields inherit DOWNWARD (a child borrows a parent's value, opt-in), but contributors aggregate UPWARD — authorship lives per-note (`lastEditor`, already tracked), so a level's contributor list is the *union bubbling up*: object's = its notes' editors; exhibit's = union across its objects; library's = union across exhibits. Resolution: **derived-by-default (auto-aggregate distinct `lastEditor`) + manual additions** (for non-annotating contributors — photographer, translator, funder), **projected to `metadata` as a "Contributors" `{label,value}` pair** — NOT `provider` (that's the publishing *institution*: Beinecke, COMPOST), and **explicitly kept OUT of the opt-in cascade** (it's an aggregate, not an inherited field — do not reuse the rights inherit-widget for it). Reinforces the collaborative-Readings story. **Build scope:** additive (after core `requiredStatement`+`rights`); the derive step is a pure projection over the heads, the manual list a small `string[]` on the level meta.
- **Viewer display (Q5, locked 2026-05-27) — one quiet credit line per view-level + an ⓘ disclosure for the rest.** IIIF makes `requiredStatement` a **MUST-display**, but the "curator's study" aesthetic forbids a shouting license banner. The full **license + `metadata` + `provider`** tuck behind an **"About / Rights" ⓘ disclosure** at each level, so the MUST-display credit is always visible while license/metadata don't clutter the read. **The Viewer reads ALREADY-RESOLVED values** — the opt-in cascade collapses at publish, so the Viewer never re-runs inheritance; it just renders whatever the published Collection/Manifest/Canvas carries. **Build scope:** the credit line ships with core (`requiredStatement`); the ⓘ disclosure is additive alongside `metadata`/`provider`.
- **Studio editing surfaces (Q6, locked 2026-05-27) — ONE shared `RightsEditor`, three placements, core-first.** The fields are identical at every level, so it is **one reusable component parameterized by level** (project one-source-of-truth rule), NOT three bespoke forms — embedded at the three placements locked in Q2: **LibraryHome** (Library) · **exhibit-overview sidebar** (Exhibit) · **object-editor sidebar beside the WADM form** (Object). **Core-first form = credit + license only:** `requiredStatement` (textarea; default label "Attribution") + `rights` (the approved-URI license picker). `provider` / `metadata` / contributors / the inherit affordance are the **additive phase**, tucked behind a **"More fields" disclosure** (symmetric with the Viewer's ⓘ). **The one invented, comprehension-gated interaction = the opt-in inherit affordance:** a **per-field "↰ Use {parent}'s value" toggle** — ON shows the parent's *actual* value greyed/locked with a pencil to override; OFF edits this level's own value (Exhibit borrows Library, Object borrows Exhibit; Library has no parent → no toggle). Chosen over a tri-state select (lighter for the common case) and over blank-means-inherit (rejected — that is the silent/automatic drift the Q2 lock forbids). ### Geo-annotation extension UX (2026-06-18 grilling)

Stress-tests the new affordances the **Map** medium (see Language) introduces, against the existing object rail / gesture model (ADR-0011) / readings rail / margin column / narrative spine. Companion to `docs/geo-annotation/DESIGN.md` (technical design) + ADR-0015 (map extent). Each affordance below is **invented UX** (philosophy #6) → earns a prototype-validation gate before lock.

- **Adding a map = a curated, modal flow (Q3).** "Map" is a medium choice on the add menu (image / audio / video / **map**); choosing it opens a **modal**: (1) pick a **basemap from a small curated provider list**; (2) set the **extent** by drawing a box on a whole-world locator — the ADR-0011 draw gesture, so "draw a box to carve out your region" *is* setting `tileSource.bounds` (Q2/ADR-0015); (3) zoom range inferred from the box. An **"advanced: custom tile URL"** escape (tier c) serves power users. The descriptor (`template`/`bounds`/`zoom`) stays **hidden** (philosophy #5 — never expose tile-URL grammar). **The curated list is load-bearing for legality:** it may contain ONLY basemaps whose terms permit static-site embedding, with attribution baked in — this resolves **D6** (the attribution/terms prerequisite) structurally, not by trusting the author. **Place-search/geocoding deferred** (an author-time external-geocoder dependency; draw-a-box ships first).
- **Geo-Notes = Box/Outline only, NO pins (Q4, 2026-06-18 user decision).** A geo-annotation is the existing **Box** (rect) or **Outline** (polygon) drawn on the map — *not* a point/pin. This **deletes** the prototype's whole pin surface (the `pin` DrawTool, the synthesized-click gesture, the constant-screen glyph — DESIGN.md T7/T8) and reuses the unchanged ADR-0011 draw gesture + Q-1 shape vocab. `GeoAnchor` = `bbox | polygon` (drop `point`).
- **Identity / anchor-of-truth = geography, not pixels (Q4, vein-4).** What makes a geo-Note *that* note is its **place on Earth**. The `archie:geo` lng/lat (a bbox for Box, a polygon for Outline) is **persisted as the source of truth**; the image-pixel selector is **derived** from it (`lngLatToPixel` into the bounded extent) for rendering and re-derived whenever the extent (ADR-0015) is re-framed. Merge reconciles on geography. **This corrects the prototype**, which stored pixels as truth and derived lng/lat — and it requires threading `archie:geo` through the spine (NewNote → AnnotationRecord → serialize), the plumbing the prototype deferred. (Mirrors the ARCHIE_EMPHASIS extension-key pattern; pure WADM consumers ignore it — Q-3 three-tier interop.)
- **Coordinate readout = display-only, at the locus (Q5, 2026-06-18).** A selected geo-region shows its coordinates in the **note popover** (ADR-0006 — editing lives at the locus, not a new panel): a **Box** → geographic bounds `[W, S, E, N]`; an **Outline** → centroid + bounds. The note list keeps a compact **center-coordinate chip**. **Not editable in v1.** Deferred to Phase 2: **typed coordinate entry** + a **geo-CSV import** (lng/lat columns → geo-regions via `lngLatToPixel`, reusing the existing "import notes from CSV" on-ramp) — the dataset-driven precision path (e.g. a known table of coordinates).
- **Map chrome = attribution + locator + on-demand reader coords (Q7, 2026-06-18).** (1) **Attribution** — a small, persistent, **always-on** credit (untoggleable: a license obligation, D6), **bottom-left** of the canvas (clears the bottom-right OSD locator), identical in Studio and Viewer; per-Object (a multi-basemap exhibit shows the *current* object's credit). (2) **Locator mini-map stays ON** for maps as well as images — the reader needs orientation within the bounded extent (user, 2026-06-18). (3) **Reader coordinates** appear only inside the **opened note** (a supplementary line in the lightbox/popup), never as persistent chrome — the read view stays clean (philosophy #3, precision-in/escape-out).
- **Map granularity = per-Object; exhibit-overview-as-map deferred (Q6/D5, 2026-06-18).** A Map is one Object (it's a medium). An exhibit holds a map Object alongside others; cross-object connection (a region → its detail photo) uses the existing **⌘K linkability** (intra-Library links), not a new aggregation. "Every Object placed on one basemap" (overview-as-canvas specialized to geography) is a separate, heavier feature — **deferred**. One map Object with many geo-regions covers the everything-on-one-map case.
- **Re-framing past existing regions = persist off-frame + warn (Q8, vein-4 lifecycle, 2026-06-18).** Narrowing a Map's extent (Q2) below an existing geo-region neither deletes nor blocks: geo is truth (Q4) so the region keeps valid lng/lat, append-only (ADR-0003) keeps it, it simply isn't shown in the current frame. The author is **warned** ("N notes fall outside the framed region") with a "widen to fit" affordance. **Deceptively-simple item** — sounds like "set bounds," hides a mode interaction with append-only + geo-truth → earns an explicit test.
- **Prototype reconciliation owed:** the built prototype is pin-based with pixel-truth; it must be reworked to Box/Outline + geo-truth, and `docs/geo-annotation/DESIGN.md` (pin-centric) revised to match. The Viewer renders from the **published** path (`loadPublishedExhibit`), so a published map needs the `tileSource` **manifest emit/readback** (DESIGN.md (e)) — the live-source thread alone isn't enough. Tracked as deferred reconciliation, not silent.

## Local view loop (2026-05-26, grilling open — not yet locked)

Two distinct surfaces, resolved as serving different moments (NOT rivals):

- **Preview (in-Studio)** — the **authoring-experience** surface. A read-only render *inside Studio* of the exhibit being authored, via the shared `render-mount` over the **published projection** (run `publishLibrary`→memory, render the heads page — NOT raw working state, so what you preview == what publishes). Single exhibit (the one in the editor). Instant; no second app; sidesteps cross-origin OPFS. _Avoid_: "preview" meaning a live-working-state render (that would drift from the published artifact).
- **Local view (closing the loop)** — the authored **Library** actually rendered by the **real Viewer app**, locally, without GitHub. What "closes the annotate→publish→view loop" for a cloned-repo author. Distinct from Preview: this is the *truthful* artifact (real Astro Viewer), not a Studio-hosted approximation.

**Demo-harness finding (blocks BOTH the local loop AND real GH-Pages publish of any non-sample Library):** the Viewer renders only hardcoded sample routes (`voynich/bidar/av.astro`) + a hardcoded gallery array (its own comment: "stand-in for exhibits.json"). It cannot render an arbitrary authored Library. Making it generic — **`[slug].astro` via `getStaticPaths` from `exhibits.json`** + **gallery read from `exhibits.json`** — is the real deliverable behind "local view," and is required for remote publish to work for any real Library regardless. The local loop is the forcing function that surfaces it.

**Hard constraint on the data bridge:** Studio's Library lives in browser **OPFS** (origin-scoped); a node script (`gen-published.mts`) cannot read it, and the two dev apps (studio:5173 / viewer:4321) cannot share OPFS cross-origin. So the bridge authored→Viewer must be a **browser-side write** (FSA into the Viewer's served tree) or a **passed artifact** (zip) — never a shared store.

**Granularity (resolved 2026-05-26):** local view publishes the **whole Library tree** (preserves "Library is the unit of publication" + keeps `buildLinkIndex` cross-exhibit links resolving); *"the exhibit"* is the route you land on, NOT a separate export primitive. **A single-exhibit export is a Library with N=1** (resolved 2026-05-26) — the degenerate case, not a new artifact type: scope the publish to one chosen exhibit (a one-exhibit Library envelope) and the Gallery's single-exhibit-collapse threshold lands the visitor straight in it. (Scoping a multi-exhibit Library to one orphans any links from the kept exhibit to dropped siblings — they degrade via `brokenLinks`.) The "two exports" (exhibit vs library) are N=1 vs N=many of the *same* artifact, never two shapes. **Exclude template exhibits by default, opt-in to include** — the bundled examples (voynich/bidar/av) are Playground-tier templates, not the author's content. Machinery already exists (`DEFAULT_EXHIBITS`/`templateSlugs`/`isTemplate`; `userExhibits` filter at App.svelte:187) but `buildFullLibrary()` ignores it — today's publish ships the demos. Fix = filter through `isTemplate` at the publish boundary.

**Viewer exhibit↔library navigation = spec'd but NOT built (2026-05-26).** Current Viewer components have ZERO up-navigation (no breadcrumb/home/back; gallery→exhibit exists via cards, exhibit→gallery does not). CONTEXT's deep-link decision already specified a clickable `Project›Exhibit›Section›Object` breadcrumb (`Project` = the Gallery, "always shown + clickable") — folded into the genericity deliverable, not new design.

**Rendering model = pre-built client shell, data-driven at runtime (resolved 2026-05-26, "one smart hall").** Reject SSG-per-exhibit (`[slug].astro` + `getStaticPaths`): authored slugs aren't known at the repo's build time, so SSG would force a per-library Astro build into the publish path — impossible in-browser, CI-shaped for GH (contradicts the "push a finished tree" adapter), and rebuild-per-change for local (kills the live loop). The exhibit view is `client:only` already (OSD touches `document`), so SSG bought nothing for the reader. Instead: ONE shell built once, reads `exhibits.json` at runtime for the Gallery, fetches `{slug}/manifest.json` on navigate. The three sample `.astro` pages collapse into it. **Consequence — local-view and GH-publish become the identical code path**, differing only in WHERE the data tree is written (FSA folder vs api.github.com): the cleanest expression yet of "publish = zip-primitive + per-host adapters."

- **Routing = hash** (`#/voynich`, `#/voynich/a/<noteId>`) (resolved 2026-05-26). Zero per-host config; consistent with the existing `#/a/<id>` deep-link idiom. History-API rejected (needs a `404.html→index` SPA-fallback hack that differs per host — the host-specific fragility the zip-primitive avoids).
- **Multi-library = NO** (resolved 2026-05-26). One published site = one Library (locked frame); the Gallery is the top of the world (UX principle #4). Multiple libraries relate via cross-Library deep-links (separate sites), never a central hub/switcher. Local view = one library at a time; swap to change (mirrors Studio's single-library "Open → replace project").
- **Local-view data bridge = a "Publish locally" host adapter** (resolved 2026-05-26), sibling to the GH adapter, reusing existing primitives. **Chromium:** `FsaFilesystem` writes the `publishLibrary` tree into a folder picked once (e.g. the repo's `apps/viewer/public/published/`); the Viewer dev server hot-reloads. **Non-Chromium/no-FSA:** Save downloads `.archie.zip`, and `gen-published.mts` — generalized to read a zip via `loadLibrary(ZipFilesystem.fromZip(...))` instead of `sample-data.ts` — expands it into `public/published/` (sidesteps OPFS-unreadable-by-node by sourcing the exported zip).
- **Preview (D) render path = published projection via shared `render-mount`** (resolved 2026-05-26): `publishLibrary`→memory→render the heads page inside Studio (single exhibit, the one in the editor). A read-only toggle in the Exhibit workspace; faithful by construction (shared `render-core`), only the Astro shell chrome (Gallery, breadcrumb) goes unexercised — which the real Viewer (C) covers.
- **ADR-0002 amendment = QUEUED** (user defers ADRs to end): add a 2-sentence note that `client:only` + arbitrary authored slugs → runtime-data shell, not SSG-per-page (Astro demotes to shell-builder + island bundler).
- **Portable Viewer = open a `.archie.zip` directly in the Viewer (grilled 2026-05-27).** Driver = the **recipient-reads gap**: someone with NO Studio is handed a library and reads it (broad charter — sharing included, NOT narrowed to private/offline; user override of the narrow recommendation). **One shell, mode auto-detected** (§216): baked tree → hosted; absent → portable empty-hall (see **Viewer** term, widened). **Entry vectors v1 (all shipped together — user chose one push over keystone-first):** (1) file pick/drop; (2) **`?src=<zip-url>`** — the shell fetches a hosted zip and renders it, the only vector yielding a shareable/citable URL, pointing at a **canonical hosted Viewer instance** (one public deploy any link targets; stays within the "no server" lock per §88 — static deploy + cross-origin fetch, same category as museum IIIF; **durability commitment** — minted `?src=` links break if the host moves, so **Publish stays the citation gold path**, `?src=` is cite-with-caveats); (3) folder-open (FSA, local-dev / "manipulable files", overlaps §224). **NOT recents** — keeps us consistent with §223 (swap-to-change, no hub); "Open another library" is that swap, in persistent chrome so the single-exhibit collapse can't trap the reader (shown in hosted AND portable as of 2026-05-27 — user override of the initial portable-only rule; ADR-0008). **Read seam (media) = RESOLVED 2026-05-27 to data-provider + `blob:` URLs** (ADR-0010; spike `spikes/portable-viewer-seam/`, 16 tests green). `published.ts` gains a portable branch `loadPortableExhibit(fs, slug)` that reads the `ZipFilesystem` and mints blob URLs for `/assets/` media (rewriting `object.source` AND note-body `/assets/` tokens) + a `revoke()` lifecycle; **zero component changes** (sinks already pass `blob:`/`data:` through — `resolve.ts:41`). Service-Worker rejected for v1 (its `/repo/published/*` interception + activate-before-first-fetch race are unverifiable on a GH project site and fight "no server"); revisit S as a v1.1 hybrid ON P only if network-true deep-links into a portable library are ever required. **Media/durability:** remote-IIIF objects keep remote URLs → a zip is self-contained ONLY for embedded `/assets/`; Viewer degrades (placeholder + open-time "needs internet for some images") — and a truly durable/offline zip **requires export-side media embedding** (Studio publish, §89.1), a flagged cross-scope dependency. **Version drift:** read the schema stamp, render same/older, degrade-with-message on newer-than-known (unknown fields ignored per §86); the migration runner stays v1.1 (§90.3); read-time/in-memory, never writes back. **Pure reader — NO outbound share/"Cite this" affordance** (sharing/citation live in Studio/Publish; the Viewer only *consumes* `?src=` links authored elsewhere); the §96 cold-arrival "open it?" survives as orientation, not sharing. **Untrusted-content boundary:** the canonical host renders strangers' zips in one shared origin → defence-in-depth: existing DOMPurify (§151 — note `render-svelte/sanitize.ts` already implements this; §231's "no sanitization yet" is STALE) + strict CSP on the canonical deploy + per-library storage isolation (clear on close) + a `?src=` content-size cap.

## Flagged ambiguities

- **Object ownership — RESOLVED 2026-05-24:** Objects (and their Notes) are **exhibit-nested**, not a shared project pool. Consequence the user accepted: no reuse across Exhibits (same painting in two Exhibits = two copies that drift); "project-level annotation" therefore means a Note targeting the Library (`Collection` URI) or Exhibit (`Manifest` URI) — curatorial/about prose — never a reusable Object.
