# Narrative is an emergent reading-mode, not a picked layout: grid is the sole arrangement

- **Amends:** mx-3e9794 (Q-2, "Layout = two orthogonal axes") — the *arrangement* axis collapses to a single value (grid); `narrative` is reclassified from a spatial arrangement to an emergent reading-mode; the §50–52 arrangement/reading-mode orthogonality is now realized in the model rather than merely declared.
- **Supersedes:** CONTEXT §105 ("Layout v1 set = Single + Grid + Narrative", 2026-05-24) — there is no author-facing layout *set* any more; arrangement is fixed and reading-mode is derived.
- **Reaffirms (untouched):** ADR-0005 / mx-41997c (Section is a self-contained reading beat, **not** a tour of Notes). The narrative spine is still Section-built; nothing here re-opens that fork.
- **Corrects (stale):** mx-39d405 ("Narrative = Phase-3, not built") — narrative has shipped; that decision no longer describes reality.
- **Source:** `/grill-with-docs`, 2026-06-19 (user-gated lock). Design: CONTEXT.md §50–52, §170–171. Code anchors: `model/layout.ts:14-23`, `model/model.ts:33-42`, `store.ts:83`, `App.svelte:993`.

## Context

A `LayoutType` union (`single | grid | narrative`) had been the author-facing way to "choose how the exhibit reads." Grilling the first-time-author journey exposed that this single field was doing two incompatible jobs — the same stroad ADR-0007 found in "Layer." CONTEXT §50–52 had *already declared* the two axes orthogonal: **spatial arrangement** (one / many / side-by-side — how objects sit) and **reading mode** (prose-led vs object-led — how the visitor moves). `LayoutType` conflated them: `single` and `grid` are arrangements, but `narrative` is a reading-mode wearing an arrangement's clothes. Asking the author to *pick* one value across both axes is the conflation made concrete.

The core code had quietly known the truth all along. `resolveLayout` (`layout.ts`) already derives the type when no explicit `exhibit.layout` is set: `sections.length > 0 ? 'narrative' : objects.length > 1 ? 'grid' : 'single'`. The only thing keeping `LayoutType` an *author-facing arrangement choice* was the `exhibit.layout ??` override and the Studio surfaces that wrote it (the LayoutPicker, its header trigger, the overview chip). Remove the override and the picker, and "narrative" stops being a thing you select and becomes a thing that *is true* once you have written sections.

This reframes narrative as an **additive layer** — a peer of Notes and Readings, all three composing on the same objects — rather than a template the author commits the whole exhibit to up front. You add a section the same way you add a note; the exhibit gains a narrative by being authored, not by a mode-switch.

## Decision

Retire `LayoutType` as an author-facing arrangement choice. Concretely:

- **Grid is the sole spatial arrangement.** The `LayoutType` union loses `single` and `narrative` as *picked* values.
  - **`single` → emergent behavior.** A 1-object exhibit auto-opens its only object (back affordance suppressed) — a grid special-case on `objects.length === 1`, computed by `resolveLayout`'s existing fallback, never selected.
  - **`narrative` → emergent reading-mode.** A narrative **exists iff `Exhibit.sections.length > 0`**. It is derived, never picked. One unnamed narrative spine per exhibit = `Exhibit.sections[]` (today's shape); no new "Flow" entity is introduced.
- **The keystone — the Viewer's leading surface is a pure function of sections-count.** `0` sections → the object grid leads; `≥1` → the **narrative** leads (primary surface) with the grid reachable behind it as an **index**; back to `0` → grid again. "A narrative exists" and "the exhibit leads with it" are the **same fact** — it can never be a separate toggle the author sets. (`layout.ts` already encodes the discriminant; this decision makes it the *only* source.)
- **Orthogonality holds (CONTEXT §170, untouched).** Spine (Section = sequence = IIIF `Range`) and Readings (interpretive lens = `AnnotationPage`; Cipher/Hoax) remain orthogonal and **compose**. A Section carries no reading ref; the active-Reading radio filters notes at whatever object the spine has framed, and the Reading legend rides in persistent chrome while reading.
- **Studio: no picker.** The "Exhibit narrative" panel (`NarrativeEditor`) is **always present** in the object-editor sidebar (ungate the `{#if currentLayout==='narrative'}` wrapper at `App.svelte:993`); its "＋ Add to the narrative" button creates beat #1. Discoverability = the always-present card + its empty-state copy (mirrors the always-mounted ReadingsRail idiom). The overview also surfaces the spine + a "Start the narrative" invitation, but beat-authoring (camera framing) happens inside the object.
- **Terminology kept, framing dropped.** "Narrative" (the whole spine) and "Section" (a beat) survive as the author-facing words — never "Reading" (reserved for the interpretive lens, §42). What is dropped is the "choosable layout template" framing. Any surviving layout chip *passively reflects* reality ("Narrative" once `sections > 0`), never offers a contradicting choice; preferably the picker is removed entirely.

## Rejected alternatives

- **(a) Keep `narrative` as a `LayoutType`.** The status quo. Rejected: it perpetuates the §50–52 conflation — a reading-mode masquerading as an arrangement the author selects — and forces the author to declare a narrative *before* writing one, when the spine's existence already answers the question. It also leaves two sources of truth (the stored `layout` field and `resolveLayout`'s derivation) free to disagree.
- **(b) Named, multiple Flows.** A first-class "Flow" entity, plural per exhibit. Rejected on intrinsic-cardinality grounds: a **Reading** is plural because interpretations *intrinsically compete* (§170–171); a reading **path** has no intrinsic plurality — there is one spine, the order you walk the objects. Naming and multiplying it invents a distinction the domain does not carry.
- **(c) Fold discrete-vs-flow into `Exhibit.mode`.** Move the choice to a `mode` field. Rejected: `mode` is **pacing** (click-driven vs scroll-driven — slideshow, scrollytelling; the reserved field from mx-3e9794). Overloading it with the discrete/flow distinction would re-commit exactly the retired-"Layer" stroad anti-pattern of ADR-0007 — one word doing two jobs. The discrete/flow fact is not pacing; it is "does a spine exist," which the spine itself answers.
- **Not the rejected ADR-0005 "tour of Notes."** This decision changes how a narrative is *entered and surfaced*, not how a Section is *built*. The flow stays Section-built (self-contained `{ objectId, start, prose }` beats); ADR-0005 / mx-41997c is reaffirmed, not reopened.

## Consequences

- **`resolveLayout` becomes the single source of truth.** Stop persisting `layout` for narrative/single intent; the resolver **always** derives from `sections`/`objects`. Removing the `exhibit.layout ??` override is the lever; the duplicate inline enum at `store.ts:83` reconciles to the narrowed union, and the exhaustive `Record<LayoutType, …>` maps (e.g. `ExhibitOverview.svelte`, `readingFamily` in `model.ts:42`) are audited against the smaller union.
- **Migration = load-time coercion, no runner, lossless.** Legacy stored `layout` values are simply ignored; the resolver re-derives. `voynich-reading` (`layout:'narrative'` + `sections[]`) becomes just `sections[]` — the spine carries over, the narrative is emergent. Update `seed-data.ts` + `sample-data.ts`; per the `test-fixtures` rule, prefer **new** fixtures over editing shared ones where consumers exist (the voynich-readings fixture has multiple consumers).
- **LayoutPicker is removed** along with its triggers: the mount (`App.svelte:1160-1164`), header trigger (`:908`), overview chip (`ExhibitOverview.svelte:159`), `setLayout` (`:456-459`), and `layoutPickerOpen` (`:436`). Routing predicates that keyed on layout (`App.svelte:185`, `hasOverview` at `:445`) rewire to **sections-based** checks. The overview trigger drops its "OR narrative" clause — `objects.length > 1` alone opens the overview (§56: sections are authored in the object editor, not on the lay-it-out screen).
- **The Viewer must keep the grid reachable as an index when a narrative leads** — this deliberately breaks today's whole-exhibit takeover (`ExhibitView.svelte:196`): the narrative leads, but the media grid degrades to a reachable list/index rather than being replaced. The 1-object auto-open path (`ExhibitView.svelte:51-54, :220`) is the `single`-as-behavior case.
- **New author-facing legibility lives in the always-present narrative card**, not a picker — its empty-state copy carries the discoverability that the removed picker used to. The keystone gets a **matched-pair cue**: a one-time, non-blocking "this exhibit now opens with your narrative … (remove every section to go back to a plain grid)" on the 0→1 transition, and its reverse on removing the last beat. This is invented UX (comprehension-gated per UX philosophy #6) and ships behind the mechanical retirement above.
- **Reducibility (drives the build phases).** *Phase 1 (mechanical, testable, low-risk):* narrow the union + reconcile `store.ts:83`; make `resolveLayout` always-derive; ungate `NarrativeEditor`; remove LayoutPicker + triggers; rewire routing predicates to sections-based; 1-object auto-open; audit the exhaustive maps; rewrite the tests asserting removed semantics (`layout.test.ts:14,21`; `model.test.ts:17`) + new fixtures. *Phase 2 (invented UX, comprehension-gated):* the matched-pair cues, recessed always-present card, overview invitation + spine surface, object-switcher, dimming explainer, reorder path preview, grid-index escape from inside the read, disambiguation + first-run copy.
